## 0章 クイックスタート（最初にこれだけ読む）

**何のシステムか**
人材派遣会社 APパートナーズ の **担当営業が、雇用契約書・就業条件明示書をWebで発行申請する**システム。StaffExpress（旧運用）の発行・署名管理を廃止し、クライアントCSVを活用してオンライン完結させる。発行→SSC承認→従業員署名→保管までをWeb上で完結し、月額費用¥0で永続運用する。

**いま動いているもの（本番）**
- 申請ウィザード（`/apply`）が本番稼働。担当営業がスタッフを検索→書類種別を選択→各STEPを入力→申請（`contracts`テーブルに保存）まで実装済み。
- SSCダッシュボード（`/dashboard/ssc`）・SSC確認画面（`/dashboard/ssc/contracts/[id]`）が本番稼働（2026-07-01実装）。
- スタッフ／部門マスタ、5システム分の個別契約CSVは本番DBに投入済み。
- CSV検索→STEP2/STEP3の自動反映ロジックは実装済み（e-staffingのみ実機確認済み、他3+Staffiaは未テスト）。

**次にやるべきこと（フェーズ別・確定）**
1. **【フェーズ1・完了】SSCダッシュボード一覧＋SSC確認・承認・差し戻し画面**（2026-07-01実装完了）
2. **【フェーズ2】担当営業ダッシュボードの中身・差し戻し再申請フロー**
3. **【フェーズ3】管理部ダッシュボード**

### 1-6. 認証統合計画（staff_roles）※2026-07-01 決定

**背景**：担当営業・SSC・管理部も派遣スタッフと同様に雇用契約書の署名者になりうるため、将来的に全員を `staff` テーブルで一元管理する方針に変更。現在はSupabase Auth（社内）とstaffテーブル独自認証（スタッフ）の2系統が併存しているが、段階的に統合する。

**新設予定テーブル：`staff_roles`**
```sql
CREATE TABLE staff_roles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  staff_id UUID NOT NULL REFERENCES staff(id),
  system_role TEXT NOT NULL CHECK (system_role IN ('担当営業', 'SSC', '管理部', '社内承認者')),
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(staff_id)  -- 1人1ロール（兼任なし。社内承認者は管理部機能も兼任する上位ロールとして設計）
);
```
- `staff`テーブル本体とは別テーブルにする理由：`staff`はStaffExpressからの定期インポートで上書きされるため、ロール情報をそこに持たせるとインポートのたびに消える。
- 兼任（1人が複数ロールを持つ）は運用上発生しないため、UNIQUE制約で1人1ロール。ただし「社内承認者」は管理部の機能も使える上位ロールとして扱う。

**ロール定義**
| ロール | できること |
|---|---|
| 担当営業 | 申請する（現場・社内問わず） |
| SSC | 現場（派遣）の申請を承認する |
| 管理部 | 管理部ダッシュボードを使う |
| 社内承認者 | 社内申請を承認する＋管理部の機能も使える。5名程度の固定メンバー |

**社内申請フロー**
- 就業場所区分が「社内」の場合：SSCをスキップして社内承認者のみが承認可能
- 申請者：担当営業または管理部
- 閲覧制限：社内社員の雇用契約書はSSCから閲覧不可（個人情報保護のため）

**ログイン画面の設計方針**
- 社内（担当営業・SSC・管理部・社内承認者）：`/login`（メール＋パスワード）
- 派遣スタッフ：ログイン画面を介さず、署名依頼メールの `/sign/[id]` リンクから直接アクセス。社員番号＋生年月日で本人確認する設計（ログイン画面の作成自体が不要）

**実装フェーズ（認証統合の順序）**
1. フェーズ1（完了）：Supabase Auth方式のままSSCダッシュボード・確認画面を実装
2. フェーズ2.5（未着手）：全ダッシュボード完成後に `staff_roles` 作成 → `/login` を独自認証（staff+staff_roles）に切替 → `sales-test`/`ssc-test`/`admin-test` の3テストユーザーを廃止 → `/apply` の認証チェックを書換
3. フェーズ5（未着手）：RLS厳格化・スタッフ署名画面（`/sign/[id]`）

> 担当営業の氏名表示（SSCダッシュボード等）は、フェーズ1では `created_by`（ユーザーID）の先頭8文字を表示。フェーズ2.5の認証統合後に `staff` テーブルの氏名表示へ切り替える。

## 3章 データベース設計（実装準拠）

### 3-1. staff（スタッフマスタ）
```sql
CREATE TABLE staff (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  employee_number TEXT NOT NULL UNIQUE,      -- 社員番号（半角数字6桁、6桁未満は前に0を付与）
  name TEXT NOT NULL,                        -- 氏名（漢字）
  name_kana TEXT,                            -- 氏名（カナ）
  dept_no INTEGER REFERENCES department_master(dept_no), -- 部門NO（部門名は表示時に結合取得）
  contract_type TEXT CHECK (contract_type IN ('有期契約','無期契約','正社員','アルバイト')),
  hired_at DATE,                             -- 入社日
  birthday DATE,                             -- 生年月日（スタッフログイン用）
  retired_at DATE,                           -- 退職年月日
  retirement_scheduled_at DATE,              -- 退職予定日（退職届未回収ケース用）
  email TEXT,                                -- メールアドレス（※テスト中はito@appart.co.jpに固定）
  crew_code TEXT,                            -- winworksの10桁スタッフコード。対象外は空欄
  password_hash TEXT,
  is_initial_login BOOLEAN NOT NULL DEFAULT TRUE,
  work_place TEXT CHECK (work_place IN ('現場','社内')),
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
-- RLS：認証ユーザーのみ参照可。INDEX：employee_number / name / name_kana / crew_code / dept_no
```
- **dept_noは「部門名(文字列)」ではなく部門マスタ参照に変更済み**（部門名変更に強くするため。表示時に `department_master` と結合）。
- **退職者除外ルール**：`retired_at` または `retirement_scheduled_at` が**検索した瞬間の日付**より前のスタッフは検索結果から除外。実装は **Supabaseクエリではなくクライアント側フィルタ**（PostgRESTの `.or()` 複数チェーンの信頼性懸念を回避するため）。

### 3-2. department_master（部門マスタ）

`id, system_type CHECK IN('e-staffing','HRstation','winworks','Staffia'), file_name, total_rows, new_rows, skipped_rows, pending_rows, error_rows, uploaded_by, uploaded_at`。

### 3-5. csv_raw_data（個別契約CSV生データ）
検索用カラムのみマッピングし、詳細はJSONBに全列保存する設計。

- STEP2/3の詳細項目は検索結果選択時に `raw_data` から `extractCsvFields` で取り出す。

**警告レベル（2段階）**
- 🔴 **赤（要確認・強）**：発行・申請は可能。SSC確認画面では通常の「承認」ボタンではなく、**「強制承認（理由入力必須）」**に切り替わる。なぜ赤警告があるのに承認するのか、理由を入力しないと承認できない。
